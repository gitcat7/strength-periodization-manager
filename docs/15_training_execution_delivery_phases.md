# 训练执行闭环：分阶段交付需求

## 来源与目标

本需求汇总自“需求方”的教练视角走查及已确认的训练执行闭环规格。产品目标是让新手到初级用户以最少操作完成：知道今日练什么、准确记录每一组、在训练后得到可信的下一次建议。

本文件只定义交付顺序。每一阶段完成并通过验收后，才发放下一阶段给执行官。

## 已完成或不在本序列内

- 审核动作库与不可读外部动作拦截已在 P0 工作中处理；本序列不得新建第二套动作目录或复制完整第三方数据。
- 自由训练完成成功页、月历历史和历史定位按照各自已下达任务推进；本序列仅复用其记录与历史入口。
- “今天时间不够”自动删减动作暂缓；当前阶段不把时间不足解释为训练失败，也不改写当日计划。
- 饮食、社交、挑战、动作视频、复杂自由模板、完整周期自动顺延不在范围内。

## 阶段 1：训练记录真实性与建议安全门槛

### 目标

防止“按计划完成”写入不真实的力量训练数据，防止主项在未完成或高强度后被错误建议加重。

### 范围

- 力量工作组点击“按计划完成”只填入计划重量和次数，不能自动伪造 RPE。
- 用户必须在当前组选择或输入合法 RPE（1-10）后才能完成该力量工作组；完整路径最多为两次明确操作：选择 RPE、完成工作组。
- 有氧/恢复动作保持现有 RPE 例外，不将力量规则错误套用。
- 主项仅在所有目标工作组完成、末组 RPE 为真实数值且不高于 8 时才可给出加重建议。
- 主项有未完成组、末组 RPE 为 9 或 10、或 RPE 缺失时，不得自动建议加重；允许维持、保守降重或提示人工确认。
- 训练建议的状态文案必须反映真实状态：只有用户明确接受建议后，才可说“已更新下次训练”；待处理、已忽略和已修改均须区分。

### 验收

1. 不能用一次“按计划完成”同时伪造 RPE 并完成力量工作组。
2. 无 RPE、RPE 0、RPE 11、未完成主项、末组 RPE 9+ 均不会产生自动加重建议。
3. 完整且末组 RPE 7-8 的主项可产生符合默认增量的加重建议。
4. 完成组、RPE、建议状态刷新后仍能在当前训练和历史中一致显示。

## 阶段 2：今日训练聚焦总览

### 目标

训练者进入今日页面后，无需在长动作表中寻找下一步。

### 范围

- 首屏展示唯一当前可执行训练的训练方向、周期位置、预计时长、动作数与主项。
- 训练过程中默认突出当前动作和下一动作，展示当前序号、已完成组/总组数、计划处方及上次工作组表现。
- 其余动作折叠到“展开全部动作”；用户仍可编辑已完成动作。
- 当前动作完成后，焦点平滑切换到下一动作，不丢失草稿或计时状态。
- 无训练日显示恢复/下一节训练说明，不默认跳到自由动作搜索。

### 验收

1. 用户可在首屏确认今天的训练方向与主项。
2. 当前训练只默认展示当前和下一动作；展开后可看全部动作。
3. 完成当前动作后焦点切到下一动作，已记录数据不丢失。

## 阶段 3：快捷组记录、休息与可靠草稿

### 目标

让普通工作组在十秒内完成记录，并让中断、刷新或短时断网不丢记录。

### 范围

- 每组提供“按计划完成”、重量/次数偏离编辑和 RPE 快捷选项 7、8、9+，同时保留精确 1-10 输入。
- 完成一组后自动启动已有休息计时，并保留跳过、增加 30 秒、关闭功能。
- 每一次组修改或完成都更新用户本地草稿；联网可用时再同步到现有服务端记录。
- 本地保存但云端同步失败时，明确显示“已保存在本机，待网络恢复后同步”，不得误报已保存到云端。
- 刷新或重新进入当前训练时恢复本地草稿；同步成功后只提交一次，不创建重复组记录。

### 验收

1. 常规力量工作组在选择 RPE 后可用一次点击完成。
2. 完成组后休息计时正确启动，且可跳过或加时。
3. 刷新后草稿恢复；模拟同步失败时数据仍存在并显示正确状态。
4. 网络恢复后不生成重复的 set log。

## 阶段 4：训练总结、建议采纳与历史一致性

### 目标

把完成记录转化为一条可理解、可追溯的下一次训练决策。

### 范围

- 训练完成总结展示完成组数、时长、训练量、主项实际表现和末组 RPE。
- 总结仅强调一条最重要的下次建议，包含建议重量/次数与简短原因。
- 用户可选择接受、修改后接受或稍后决定；选择与建议一起写入历史。
- 用户编辑历史组数据后，重新计算训练量、完成度和仍待处理的建议；已接受/已修改/已忽略的历史决定不被悄悄覆盖。
- 结束后可以回首页或经既有 URL 查看历史详情。

### 验收

1. 每次完成训练都有可读总结和最多一条重点建议。
2. 建议的三种用户选择均持久化，并在历史中正确展示。
3. 修改历史后，训练量和待处理建议重新计算；已决建议保持其用户决定。
4. 自由训练不自动篡改周期计划，但可以写入历史和进展。

## 发布门槛

- 每一阶段先补失败测试，再实现最小改动；阶段提交独立、可回滚。
- 每阶段结束至少运行相关测试；阶段 1 和阶段 4 额外运行 `pnpm test`、`pnpm lint`、`pnpm typecheck`、`pnpm build`。
- 需要登录态的行为以浏览器烟测验证，覆盖计划训练、恢复日、自由训练、历史编辑和断网恢复模拟。
- 执行官不得自行部署或合并；每阶段报告提交 SHA、文件、验证结果、遗留风险和下一阶段是否可开始。
