# wger 外部动作引用设计

## 目标

将完整动作目录从项目静态文件和 Supabase 本地动作表中移出，改为通过 wger 的公共 REST API 引用；保留周期算法所需的最小本地核心动作定义，避免重写推拉蹲计划、调重建议和 PR 逻辑。

## 已确认范围

- 产品采用混合引用：完整动作浏览和单次训练选择使用 wger；周期算法只保留最小本地核心动作档案。
- 不同步或持久化完整第三方动作目录到 Supabase、仓库静态 JSON 或浏览器长期缓存。
- 单次训练使用外部动作引用，训练完成时保存动作名称和关键元数据快照。
- 不直接由浏览器请求 wger；请求统一经过本项目的服务端适配器。
- kg 是唯一重量单位，继续遵守现有用户隔离、RLS 和移动端优先规范。

## 非目标

- 不接入 wger 的用户、计划、饮食、体重或训练日志 API。
- 不复制 wger 动作图片、视频或说明文本到本项目。
- 不在本阶段把全部周期计划动作改成外部引用。
- 不因上游不可用而阻止用户查看已有训练记录。

## 架构

```text
移动端动作搜索
  -> Next.js /api/exercise-catalog/*
  -> wger 公共 API
  -> 标准化动作引用

单次训练完成
  -> provider + external_exercise_id + 名称/元数据快照
  -> workout_exercises / set_logs
  -> 历史、进展、导出
```

### 服务端 wger 适配器

- 仅访问公开动作查询端点；不保存、暴露或要求 wger 用户令牌。
- 提供搜索、按部位筛选和动作详情的稳定内部接口。
- 标准化为 `provider`、`externalId`、`name`、`description`、`muscles`、`equipment`、`category`、`sourceUrl`。
- 使用短时内存或边缘缓存降低上游请求量；缓存失效后重新请求，禁止写入 Supabase 动作目录。
- 上游超时、非 2xx、分页异常或格式变更时返回可读错误；单次训练编辑页保留已选动作，不丢失用户输入。
- 明确限制响应大小、分页上限和超时，避免上游慢响应拖慢 Vercel 函数。

### 本地核心动作边界

- `exercises` 仅保留推拉蹲计划生成、重量建议、PR 与历史兼容所需的最小核心档案。
- 这些档案不是完整动作库，也不再承担动作浏览搜索职责。
- 原有 `hasaneyldrm/exercises-dataset` 静态目录、加载器、同步脚本和界面来源说明将在迁移后移除。
- 周期计划的核心动作详情可展示 wger 来源链接，但计划算法不依赖 wger 在线可用性。

## 数据模型

为单次训练的动作记录新增或迁移为以下外部引用字段：

- `exercise_provider`: 固定值 `wger`。
- `external_exercise_id`: wger 运动动作的稳定外部 ID。
- `exercise_name_snapshot`: 完成时的显示名称。
- `exercise_metadata_snapshot`: JSONB，仅保留部位、器械、来源 URL 等显示/归因所需的最小元数据。

`workout_exercises.exercise_id` 对单次训练允许为空；周期计划记录继续要求本地核心动作 ID。数据库约束必须保证每条动作记录恰好满足以下之一：

1. 周期计划动作：本地 `exercise_id` 有值，外部引用为空。
2. 单次训练外部动作：`exercise_provider`、`external_exercise_id` 与名称快照均有值，本地 `exercise_id` 为空。

所有查询、RLS、历史、进展、导出与 Agent API 必须处理两种形态。完成时快照是历史事实，不会随 wger 后续编辑而改变。

## 页面与交互

### 动作浏览

- 搜索默认跨分类，支持中文或英文名称的文本匹配；分类是辅助筛选，不应遮蔽搜索结果。
- 只显示文字信息：名称、主要部位、器械；不使用第三方图片。
- 动作详情显示精简说明、来源标识和跳转 wger 的外部链接。
- 无网络或 wger 不可用时，显示重试入口与明确状态，不伪造本地目录结果。

### 单次训练

- 从 wger 结果添加动作，保存 `provider/externalId`，不创建本地完整动作条目。
- 草稿和完成记录均可恢复已选动作和组数据；完成后锁定快照。
- 有周期计划时，单次训练不影响计划下一节、计划重建或训练建议的作用范围。

## 许可与归因

- 界面设置/隐私或动作详情处提供 wger 来源与数据许可链接。
- 不复制 wger 应用代码；仅按其公开 API 使用数据。
- 上线前逐条核对所消费字段对应的动作数据许可，避免把 AGPL 应用代码许可与动作数据许可混为一谈。

## 验收

1. 项目不再打包或从 Supabase 读取完整第三方静态动作目录。
2. 动作搜索、筛选和详情经内部适配器可用，且浏览器不直接请求 wger。
3. wger 不可用时，用户输入和已选动作保持不丢失，页面给出明确的重试状态。
4. 单次训练保存的外部 ID 与名称快照可在历史、进展和 CSV 导出中正确显示。
5. 周期计划、推荐、PR 与既有训练历史继续使用本地最小核心动作，行为不回归。
6. RLS 确保用户只能读写自己的外部动作训练记录；服务端代理不接受用户提供的 URL。
7. 覆盖 API 适配、字段标准化、上游异常、外部动作草稿/完成、快照稳定性、周期计划隔离和迁移约束的测试。
