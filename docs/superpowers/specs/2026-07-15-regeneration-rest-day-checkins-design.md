# 计划重建交互与休息日打卡设计

## 目标

让计划页的“重新生成计划”成为可预览、可确认、可恢复的操作；让排程中的休息日成为可见、可一键打卡的正式日程，而不是训练日期之间的空白。

首版休息日只支持一键完成，不收集睡眠、疲劳、酸痛或文字备注。

## 已确认范围

- 重新生成前必须显示确认抽屉。
- 用户可以看到新计划的训练模板、排程、起止日期、训练日和休息日数量，以及旧计划中将被替换的未完成训练数。
- 已完成训练、已完成休息日、历史训练记录、PR 与进度数据必须保留。
- 练 N 休 M 排程必须生成显式休息日；练一休一显示为“训练日、休息/恢复日、训练日”的连续日程。
- 固定星期排程在计划起止日期内的非训练日也生成显式休息日。
- 自由排程没有可确定的日历日期；首版不生成休息日记录，并在界面说明“按完成后自行安排恢复”。
- 休息日可在今日页一键完成，并在计划页、历史页显示已完成状态。
- 休息日绝不计入训练量、e1RM、PR、动作推荐或力量训练完成率。

## 现状与问题

现有 `cadence` 计算会在训练日期之间跳过休息天，但只生成训练 `workouts`。因此计划页没有休息日，今日页没有可打卡对象。

现有重新生成流程先归档全部活动计划，再通过多次客户端写入创建新计划。任一后续写入失败时，用户可能没有活动计划。新的确认交互不能只补一个弹窗，必须同时解决这项状态一致性风险。

## 数据模型

将 `workouts` 扩充为统一的计划日程表：

- `day_type text not null default 'training' check (day_type in ('training', 'rest'))`
- `schedule_index integer not null`：同一计划内按日期连续递增，训练日和休息日都占用。
- `sequence_index integer null`：仅训练日使用，保持训练推进顺序；休息日为 `null`。

迁移要求：

1. 现有训练记录回填 `day_type='training'`，`schedule_index=sequence_index`。
2. 将 `sequence_index` 改为可空；保留 `(program_id, sequence_index)` 唯一约束，利用 PostgreSQL 对多个 `null` 的允许来兼容多个休息日。
3. 新增 `(program_id, schedule_index)` 唯一约束与 `(program_id, scheduled_date, schedule_index)` 查询索引。
4. `completed_at` 继续作为训练与休息日的完成时间；不新增休息日志表。
5. RLS 保持以 `workouts.user_id = auth.uid()` 为边界；休息打卡只能更新本人、`day_type='rest'`、未完成的日程。

训练日依然拥有 `workout_exercises`；休息日禁止创建任何 `workout_exercises`。数据库约束或 RPC 校验必须阻止向休息日插入动作。

## 排程规则

### 练 N 休 M

先根据模板生成训练日处方和训练 `sequence_index`，再从第一天开始展开为日程：

- 每生成 `trainDays` 个训练日，插入 `restDays` 个 `day_type='rest'` 日程。
- 训练日的 `sequence_index` 连续递增；休息日为 `null`。
- `schedule_index` 对所有日程连续递增。
- 周期最后一个训练日后不追加无意义的尾部休息日。

例如推/拉/蹲模板使用练一休一时：

| 日程 | 类型 | 训练序列 |
| --- | --- | --- |
| 推 A | training | 1 |
| 休息/恢复日 | rest | - |
| 拉 B | training | 2 |
| 休息/恢复日 | rest | - |
| 蹲 A | training | 3 |

### 固定星期

在第一节训练日至最后一节训练日期间，选中星期生成训练日，其余日期生成休息日。训练日仍按模板顺序分配 `sequence_index`。

### 自由排程

保持当前“建议日期不固定”的行为，不生成虚假的日历休息日。计划页显示恢复提示，但不提供休息打卡。

## 重新生成交互

点击活动计划区域的“重新生成计划”后，不立即执行生成：

1. 打开移动端底部抽屉、桌面端居中对话框。
2. 抽屉展示当前选择的模板与排程摘要、新计划日期范围、训练日数量、休息日数量，以及旧计划未完成训练日数量。
3. 明确提示：“已完成记录会保留；当前计划中未完成的训练与休息日将归档，不再显示为当前日程。”
4. 次操作为“继续编辑”，主操作为“确认生成新计划”。
5. 提交期间锁定表单和关闭操作，主按钮显示加载图标与“正在生成计划”。
6. 成功后关闭抽屉、清理训练缓存、重新加载活动计划、滚动到新计划首个日程，并显示成功提示。
7. 失败时保留抽屉和选择内容，显示安全中文错误；旧活动计划仍可继续使用。

## 原子重建

新增认证 RPC `replace_active_program`，以当前认证用户为唯一身份来源，接收已在客户端计算的受限 JSON 计划载荷。

在一个数据库事务中：

1. 校验模板、排程、日期、日程类型、索引连续性、重量和动作处方。
2. 插入新 `programs`、所有训练/休息日程、训练日的动作处方。
3. 将该用户此前 `active` 计划更新为 `archived`。
4. 激活新计划并返回新计划 ID、首个日程 ID、训练日/休息日数量。

任意一步失败则整个事务回滚，旧活动计划不变。客户端不再以“先归档、后逐条写入”的方式重建计划。

## 今日页与打卡

今日查询分为两个独立结果：

1. 若当天存在未完成的 `day_type='rest'` 日程，显示“今日恢复”卡片。
2. 下一节力量训练始终只查询 `day_type='training'` 的最小未完成 `sequence_index`，休息日不会阻塞训练推进。

恢复卡片包含日程日期、简短恢复提示和一个主按钮“完成休息”。点击后仅更新该日程为 `completed` 并写入 `completed_at`；成功后展示对勾和“今日恢复已完成”。如有待继续的力量训练，显示次级链接“查看下一节训练”。

训练日界面、动作日志、组间计时器、动作替换和 Coach 逻辑保持不变。

## 计划、历史与进度

- 计划页按 `schedule_index` 呈现所有日程；休息日用低强调恢复样式、月亮图标和状态标记，训练日保留原有方向与强度信息。
- 计划摘要分别展示“训练日”和“休息/恢复日”数量。
- 历史页显示已完成休息条目，但不渲染动作、组数、容量、RPE 或编辑入口。
- 进度、CSV、PR、推荐和完成率查询明确过滤 `day_type='training'`。
- “下一节训练”与首页连续性计算只基于训练日；休息打卡可作为独立恢复完成指标展示，但不混入力量训练完成率。

## 事件与缓存

新增埋点：

- `program_regeneration_opened`
- `program_regenerated`
- `rest_day_completed`

重建成功清除训练、计划、今日和草稿缓存；休息日完成清除今日与计划缓存。所有事件只包含模板、排程、训练/休息日数量和匿名化日程 ID，不写入用户 ID、邮箱或训练明细。

## 验收

1. 领域测试覆盖练一休一、练三休一、固定星期、自由排程；训练与休息日顺序、日期、索引均正确。
2. 数据库测试覆盖旧数据回填、日程约束、休息日无法含动作、RLS、RPC 原子回滚与用户隔离。
3. 重新生成抽屉在未确认前不写数据库；成功重建后旧计划归档、新计划激活；模拟任意插入失败时旧计划仍为活动状态。
4. 今日页能显示并一键完成休息日；待完成休息日不会阻塞下一节力量训练。
5. 计划和历史显示休息日；进度、PR、CSV 容量与力量完成率不受休息打卡影响。
6. 320px、390px、768px、1440px 下抽屉/对话框、休息卡片和计划列表无重叠、无横向溢出，键盘可关闭并恢复焦点。
7. 完整类型检查、单元测试、数据库测试、构建与本地烟雾测试通过。

## 非目标

- 不采集睡眠、疲劳、酸痛、心率或恢复备注。
- 不自动根据休息打卡改变负重、训练顺序或 PR 日期。
- 不在自由排程中伪造固定日历休息日。
- 不重写既有训练模板、动作库或 Fitness Coach 处方规则。
