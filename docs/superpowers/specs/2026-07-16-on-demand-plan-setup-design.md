# 按需填写计划参数设计

## 目标

移除登录后必须先创建“训练画像”的流程。用户登录后可以立刻使用首页、单次训练、历史、进展、动作库和 PR；只有在首次创建或重新生成周期计划时，才在计划页填写生成计划真正需要的最少参数。

## 用户体验

### 登录和旧链接

- 邮箱登录默认回跳 `/`，不再以 `/onboarding` 为默认目的地。
- 仍携带合法 `next` 参数的登录链接必须优先回到该页面。
- `/onboarding` 保留为兼容路由，并立即跳转到 `/plan`；不再展示或保存独立训练画像表单。
- 首页、设置和诊断页移除“创建训练画像”“修改训练画像”入口与文案。

### 计划页

- 没有可用计划配置或用户点击“重新生成计划”时，计划页展示计划参数表单或其对话框，而不是提示用户先去训练画像页。
- 表单的必填项：每周训练天数、可训练日、主要目标、单次时长，以及至少一个主项的最近稳定工作组（重量 kg 和次数）。
- 表单的可选项：训练经验、伤病/禁忌动作。
- 用户提交后，在同一操作中保存计划配置并生成四周计划；成功后直接展示计划结果。
- 已有用户的训练参数可以作为表单初始值，并可在重新生成计划时修改。
- 单次训练不读取、创建或要求计划参数，用户可以完全不建计划而记录一次训练。

### PR 页面

- 当用户为目标动作缺少最近工作组时，PR 页不再引导到 `/onboarding`。
- 该页面应提供就地补充入口：填写该动作的重量 kg 和次数，保存为该动作的最近工作组后继续创建或评估 PR 目标。
- 不要求用户补全其他动作或完整计划参数。

## 数据与兼容性

- 保留 `athlete_profiles` 和 `lift_profiles` 表、现有 RLS 以及已有用户数据；不删除、不清空、不迁移历史记录。
- 两张表在产品语义上改称“计划参数”和“主项最近工作组”，不再面向用户显示“训练画像”。
- 计划生成继续从这些表读取参数，避免改写已验证的推拉蹲计划、调重、PR 和教练规则。
- 首次建计划时在现有写入路径中创建/更新相应记录；重新生成时覆盖本次表单明确提交的字段。
- 至少一个主项工作组是首次生成计划的最低数据门槛。系统只为有数据的主项计算训练重量；其余主项沿用现有安全降级逻辑，绝不虚构用户能力数据。

## 代码边界

- 登录表单和魔法链接解析的默认目的地由 `/onboarding` 改为 `/`，但安全内部路径校验保持不变。
- `src/app/onboarding/page.tsx` 只负责兼容跳转；移除 `OnboardingForm` 在该路由上的产品展示。
- 计划生成组件拥有“读取已有计划参数、展示/校验表单、保存、生成”的完整流程，不能再把用户导向 onboarding。
- PR 管理组件拥有单项最近工作组补充流程，不依赖 onboarding。
- 首页、设置、诊断、今日训练、计划空状态和隐私文案全部替换用户可见的“训练画像”措辞；技术变量名称可以保留，避免无关重构。
- 不新增数据库表；只有在实现发现现有约束无法支持可选配置时，才能新增前向迁移，且必须说明原因并同步 `supabase/schema.sql`。

## 失败处理

- 计划参数提交失败时，保留用户已输入内容，显示数据库返回的安全错误摘要，并允许重试。
- 未填完必填项时在当前表单内给出字段级提示；不跳转到其他页面。
- 读取已有参数失败时，计划页显示可重试状态；不能阻止用户使用单次训练或浏览其他页面。
- PR 就地补充失败时，保留重量/次数并显示重试状态。

## 验收

1. 新用户魔法链接登录后默认进入首页，而不是 `/onboarding`。
2. 新用户无需填写任何计划参数即可创建单次训练、查看历史/进展/动作库和访问 PR 页面。
3. 新用户从计划页首次创建计划，能在一个表单完成最少参数填写并生成四周推拉蹲计划。
4. 计划页、PR 页、今日训练和首页不再出现“请先创建训练画像”或跳转 `/onboarding` 的阻塞流程。
5. PR 缺少某个动作工作组时，可以在 PR 当前页面补充该动作的 kg/次数并继续。
6. 已有用户的计划、训练记录、主项数据、PR 目标和重建计划流程仍可用。
7. `/onboarding` 老链接跳转到 `/plan`；登录链接携带合法 `next` 时仍优先回到 `next`。
8. 相关单元/组件测试、类型检查、lint、生产构建及认证后浏览器烟测均通过。
