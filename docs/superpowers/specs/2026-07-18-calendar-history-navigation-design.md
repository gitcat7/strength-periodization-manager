# 月历训练历史与导航设计

## 目标

将训练历史设为移动端一级入口，让用户能够按月、按日期回看周期训练、自由训练和恢复日，并从任意已记录日期进入完整训练详情。

## 导航

- 移动端底部导航固定为五项：`今日`、`计划`、`历史`、`进展`、`设置`。
- `历史` 指向 `/history`，使用日历图标；底部导航中不再保留独立 `PR` 项。
- PR 功能保留：首页的 PR 卡片继续进入 `/pr`，进展页提供与 PR 相关的入口或链接。不会删除 PR 页面、目标或历史数据。
- 桌面导航遵循同一信息架构，确保 `/history` 容易发现。

## 历史页：月历主视图

- 历史页默认显示当前月份，可通过上个月/下个月按钮切换；不允许选择未来月份。
- 月历以周一为每周第一天，补足月首/月尾的相邻月份日期，但相邻月份日期仅作视觉占位，不能混入当前月统计。
- 每个日期最多显示两行紧凑标签，避免手机日历格溢出：
  - 已完成训练：`训练名称或自由训练` + `训练量 kg`。
  - 已完成恢复日：`恢复` 标签，不展示虚构吨位。
  - 已计划但未完成训练：低强调度展示训练方向，例如 `推 A`；不展示完成训练量。
  - 没有训练：只显示日期。
- 同一天若有多个完成记录，日历格显示完成记录数量和合并训练量；点击后在日期摘要中列出全部记录。
- 日期格不使用随机颜色。状态以文字和有限的统一状态色区分：已完成、计划中、恢复日、今天。

## 日期摘要与详情

- 点击当前月任意有记录的日期，在月历下方显示该日摘要；首次打开时默认选中今天，若今天无记录则不强制显示空摘要。
- 摘要列出该日每次训练：训练类型（自由训练或计划训练）、完成时间/状态、完成组数、训练量和最高 e1RM（不适用时显示“不适用”）。
- 点击某条摘要进入现有训练详情，URL 使用 `/history?workout=<id>`；详情页需默认展开并高亮该 workout。
- `workout` 无效或不属于当前用户时，历史页正常显示用户自己的月历与列表，不泄露第三方记录。
- 从“完成训练成功页”的“查看本次记录”和首页“最近一次训练”均使用这条 URL 进入同一详情，不重复建设详情页。

## 数据边界

- 读取现有 `workouts`、`workout_exercises`、`set_logs` 和用户自己的自由训练记录；按本地日期归档。
- 自由训练已完成记录计入月历、摘要、历史、进展与 CSV，但不会改变周期计划的下一节、完成状态或重建逻辑。
- 周期训练、自由训练和恢复日必须清楚区分；所有数据继续由当前 Supabase RLS 保护。
- 月历仅请求当前展示月份和必要摘要字段，避免一次读取全部历史记录。

## 空状态与失败处理

- 当前月没有记录时显示“本月还没有训练记录”，提供“记录自由训练”和“查看训练计划”入口。
- 加载失败显示可重试状态；保留已加载月份和已选日期，不显示错误的零训练量。
- 月份切换加载中禁用重复切换按钮并保持上一个月内容，避免页面空白闪烁。

## 验收

1. 底部导航仅有五项且包含历史，PR 仍可从首页和进展进入。
2. 历史页可按月切换并按日期显示完成训练、计划训练与恢复日。
3. 自由训练与计划训练均可在对应日期看到，且训练量数据正确。
4. 点击日期可看到该日摘要；点击摘要可定位并高亮该条历史详情。
5. 无效或非本人 `workout` 参数不泄露任何记录。
6. 完成训练成功页与首页最近训练均能进入同一历史详情路径。
7. 手机宽度下日期标签、底部 Tab 和摘要文字不重叠、不溢出。
8. 相关单元/组件测试、类型检查、lint、生产构建和登录后的浏览器烟测通过。
