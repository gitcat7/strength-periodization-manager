# MVP 可行性评审、资源清单与排期

版本：v0.1  
日期：2026-07-01  
背景：用户每天约 22:00 可协助决策、提供账号资源、验收和反馈。

## 1. 可行性评审结论

结论：**可实行，建议推进 MVP。**

推荐路线仍然是：**移动端优先响应式 Web/PWA + 托管后端服务 + 规则引擎先行**。

可行性评分：

- 需求真实度：8/10
- MVP 技术可行性：8/10
- 一个人运维可行性：7/10
- 商业化不确定性：6/10
- 训练建议责任风险：中等，需要免责声明和保守规则

总体判断：

这个产品适合从小而准的 MVP 做起。核心难点不在技术，而在“训练规则是否足够可信”和“用户能否连续记录 4 周”。第一版不应该追求大而全，而要验证用户是否真的愿意把训练计划交给系统动态调整。

## 2. 为什么它可实行

### 2.1 产品闭环清晰

MVP 闭环足够短：

```text
填写训练画像 -> 生成计划 -> 今日训练 -> 记录组数据 -> 自动调重 -> 下次训练继续执行
```

这比“内容社区”“AI 教练”“健身课程平台”都更容易做出第一版。

### 2.2 技术复杂度可控

第一版不需要机器学习，也不需要复杂推荐模型。核心能力可以用确定性规则完成：

- e1RM 估算
- 训练最大值 TM
- 完成率判断
- RPE 兜底
- 连续失败降载
- PR 测试日安排

这些规则可以先写成可测试的 TypeScript domain functions，后续再迭代。

### 2.3 运维可以托管化

一个人做产品，不能把时间耗在服务器维护上。使用 Vercel + Supabase + Sentry + PostHog 这类托管服务，能把大部分运维压力转成可控费用。

### 2.4 MVP 可以低成本验证

前 20-50 个内测用户可以基本跑在免费额度或很低成本上。真正需要花钱的只有域名，以及必要时升级数据库或部署套餐。

## 3. 主要风险与应对

### 3.1 用户不持续记录

风险：

健身记录工具最大敌人是用户懒得记。

应对：

- 今日训练页必须极简。
- 默认填好目标重量和次数。
- 支持一键完成目标组。
- 只在异常时让用户输入更多信息。

### 3.2 训练建议被认为不专业

风险：

进阶用户会质疑规则，新手可能过度相信系统。

应对：

- 每条建议必须给原因。
- 建议默认保守。
- 允许用户修改建议。
- 在 PR、失败、疼痛场景显示安全提示。

### 3.3 范围失控

风险：

动作库、视频、社交、AI、营养、教练端都会诱人，但都会拖慢 MVP。

应对：

- 第一版只做力量训练主项。
- 只做 3 天和 4 天模板。
- 只做一个当前计划。
- 不做复杂内容系统。

### 3.4 法务与安全边界

风险：

训练建议可能被用户理解为医疗或康复建议。

应对：

- 明确免责声明。
- 不处理伤病康复方案。
- 疼痛、伤病、异常不适时只提示咨询专业人士。
- PR 测试建议保守，避免鼓励无保护极限尝试。

## 4. 需要你准备给我的资源

这些资源不一定第一天全给，但越早给，开发越顺。

### 4.1 产品决策资源

必须准备：

- 产品名称候选 3-5 个。
- 目标用户优先级：更偏小白，还是更偏有基础训练者。
- 第一版训练模板选择：3 天全身、4 天上下肢是否都要上。
- 默认单位：kg 是否作为主单位。
- 是否允许用户自定义动作：MVP 建议先不开放复杂自定义。
- 是否需要中文名和英文名同时准备。

建议准备：

- 你自己的训练记录样本 2-4 周。
- 你认可的训练计划样本，比如 5x5、531、DUP 中你喜欢的部分。
- 你希望系统“不要做”的训练建议清单。

### 4.2 训练业务资源

必须准备：

- 核心动作列表：深蹲、卧推、硬拉、推举、划船、引体等是否确认。
- 每个动作默认加重幅度：上肢 2.5kg、下肢 5kg 是否确认。
- 新手起始强度保守系数是否接受。
- PR 测试保守策略是否接受。
- 失败后的降载策略是否接受。

建议准备：

- 你认为“训练失败”的定义。
- RPE 是否必须输入：MVP 建议可选。
- 哪些动作需要安全提示。

### 4.3 内测资源

必须准备：

- 5 个种子用户，最好包含 2 个新手、2 个初级、1 个有经验者。
- 每个内测用户愿意连续使用 4 周。
- 每周能反馈一次：计划是否合理、重量建议是否合理、记录是否麻烦。

建议准备：

- 一个微信内测群。
- 一份简单反馈表。
- 内测用户当前主项水平。

### 4.4 账号和部署资源

你需要自己注册并控制账号，不要把主密码发给我。后续只需要给我项目里的环境变量值或让我指导你配置。

必须准备：

- GitHub 账号。
- Vercel 账号。
- Supabase 账号。
- 一个域名，或先用 Vercel 免费域名。

上线前建议准备：

- Sentry 账号，用于错误监控。
- PostHog 账号，用于产品埋点。
- Resend 账号，用于正式邮件发送。
- Cloudflare 账号，用于域名和 DNS。

### 4.5 合规与文案资源

必须准备：

- 免责声明是否采用当前 PRD 里的版本。
- 隐私政策是否需要简版。
- 用户协议是否先用简版。

建议准备：

- 联系邮箱。
- 产品介绍一句话。
- 内测招募文案。

## 5. 需要购买或租赁的资源

### 5.1 MVP 最小成本组合

适合阶段：本地开发 + 5-20 人内测。

| 类别 | 服务 | 费用判断 | 渠道 | 用途 |
|---|---|---:|---|---|
| 代码托管 | GitHub Free | $0 | https://github.com/pricing | 私有仓库、Issue、基础 CI |
| 部署 | Vercel Hobby | $0 | https://vercel.com/pricing | Next.js 应用部署 |
| 数据库/Auth | Supabase Free | $0 | https://supabase.com/pricing | Postgres、登录、基础 API |
| 错误监控 | Sentry Developer | $0 | https://sentry.io/pricing/ | 单人错误监控 |
| 埋点 | PostHog Free tier | $0 起 | https://posthog.com/pricing | 产品分析、漏斗、留存 |
| 邮件 | Resend Free | $0 | https://resend.com/pricing | 登录邮件、通知邮件 |
| 域名 | Cloudflare Registrar | 按注册局成本 | https://www.cloudflare.com/products/registrar/ | 正式访问域名 |

这个组合可以先跑起来。唯一强烈建议购买的是域名，但也可以等到内测前再买。

### 5.2 推荐内测组合

适合阶段：20-100 人内测，数据不能随便丢，体验要稳定。

| 类别 | 服务 | 当前官方信息摘要 | 建议 |
|---|---|---|---|
| 部署 | Vercel Hobby 或 Pro | Hobby 为 $0/mo；Pro 为 $20/mo | 先 Hobby，遇到协作/性能/商用限制再升 Pro |
| 数据库/Auth | Supabase Free 或 Pro | Free 有 2 个免费项目；Free 数据库 500MB；Pro/Team 数据库 8GB 起 | 先 Free，正式内测或担心项目暂停再升 Pro |
| 错误监控 | Sentry Developer | Developer 免费，限 1 个用户 | 先免费 |
| 埋点 | PostHog | Product analytics 前 100 万 events/mo 免费 | 先免费 |
| 邮件 | Resend | Free 3000 emails/mo，100 emails/day；Pro $20/mo | 先免费，正式邀请大量用户再升 |
| 域名/DNS | Cloudflare Registrar | 域名按成本注册，含免费 DNS/SSL/WHOIS redaction | 建议购买 |

### 5.3 预计月成本

开发期：

- $0/月，外加域名年费。

小规模内测期：

- $0-45/月。
- 可能产生费用：Vercel Pro $20/月、Supabase 付费计划或 compute、Resend Pro $20/月。

正式 MVP 公测期：

- $45-100/月比较合理。
- 成本来自数据库、部署、邮件、监控和埋点超额。

## 6. 购买与配置顺序

第一优先级：

1. GitHub：创建私有仓库。
2. Supabase：创建项目，选择离目标用户较近的区域。
3. Vercel：绑定 GitHub 仓库，配置自动部署。

第二优先级：

4. Cloudflare：购买域名并配置 DNS。
5. Sentry：创建项目并拿 DSN。
6. PostHog：创建项目并拿 Project API Key。

第三优先级：

7. Resend：绑定域名，配置 DKIM/SPF/DMARC，用于正式邮件发送。

## 7. 你每天 22:00 需要参与什么

你的时间很宝贵，所以每天只处理三类事情：

1. **决策**：二选一或三选一，不做长会。
2. **授权/账号配置**：注册账号、复制环境变量、确认付款。
3. **验收**：打开页面，按我给的测试脚本走一遍。

建议每天 22:00 固定 20-30 分钟：

```text
22:00-22:05 看当天成果
22:05-22:15 做关键决策或提供账号资源
22:15-22:25 按测试清单验收
22:25-22:30 确认第二天目标
```

## 8. MVP 排期

按一个人协助、我负责主开发来估算，建议做 **6 周 MVP**。如果中间账号、支付、域名、需求决策拖延，可能到 8 周。

### 第 0 周：准备周，1-3 晚

目标：

- 确认产品名、MVP 范围、技术栈。
- 注册必要账号。
- 创建仓库和项目。

你需要做：

- 确认产品名称。
- 注册 GitHub、Vercel、Supabase。
- 决定是否购买域名。

交付：

- 技术项目骨架准备完成。
- 环境变量清单完成。
- 数据库设计最终版。

### 第 1 周：项目骨架与登录

目标：

- 搭好 Next.js 应用。
- 接入 Supabase Auth。
- 完成基础导航和移动端布局。

你 22:00 协助：

- 第 1 晚：确认 UI 风格和产品名。
- 第 2 晚：提供 Supabase 项目信息。
- 第 3 晚：验收登录注册。

交付：

- 可登录的线上预览地址。
- 基础页面框架。

### 第 2 周：训练画像与计划生成

目标：

- 完成 onboarding。
- 完成主项水平输入。
- 生成 3 天/4 天训练计划。

你 22:00 协助：

- 确认默认训练模板。
- 用自己的训练数据试填。
- 判断生成计划是否合理。

交付：

- 新用户能生成第一份 4 周计划。
- 周计划页可查看。

### 第 3 周：今日训练与训练记录

目标：

- 完成今日训练页。
- 完成组记录。
- 完成草稿保存。
- 完成训练总结基础版。

你 22:00 协助：

- 按真实训练流程点一遍。
- 指出记录是否麻烦。
- 确认表单默认值是否符合健身房场景。

交付：

- 可以完整记录一次训练。
- 数据可进入历史记录。

### 第 4 周：自动调重与规则引擎

目标：

- 完成 e1RM、完成率、训练量计算。
- 完成下次重量建议。
- 完成采纳、修改、拒绝建议。

你 22:00 协助：

- 提供几个真实案例：完成、失败、RPE 很高、缺训。
- 判断系统建议是否保守合理。

交付：

- 训练完成后能生成下次建议。
- 采纳后自动更新后续计划。

### 第 5 周：PR 计划、进展页、数据导出

目标：

- 完成 PR 目标。
- 完成 PR 准备阶段和测试日建议。
- 完成 e1RM 趋势和训练量趋势。
- 完成 CSV 导出。

你 22:00 协助：

- 创建一个真实 PR 目标。
- 检查测试日重量阶梯。
- 验收趋势图是否看得懂。

交付：

- 产品核心差异化功能可演示。

### 第 6 周：内测发布与运维

目标：

- 接入 Sentry。
- 接入 PostHog。
- 配置域名。
- 完成免责声明、隐私政策、反馈入口。
- 邀请第一批内测用户。

你 22:00 协助：

- 购买或确认域名。
- 提供内测用户名单。
- 发送招募文案。
- 每晚看反馈和 bug。

交付：

- MVP 内测版上线。
- 5-20 个用户可使用。
